# AGENTS.md — YellowDog Python SDK

## What this project is

`yellowdog-sdk-python` is the official Python client library for the **YellowDog Platform**, a dynamic cloud workload management and provisioning solution that automates the selection of compute resources for large-scale, computationally intensive workloads. It functions as a scheduler and orchestration engine across hybrid and multi-cloud environments, with features such as Spot Fleet support, dynamic provisioning, and cost optimization.

The SDK is published as the [`yellowdog-sdk`](https://pypi.org/project/yellowdog-sdk/) package and documented at https://docs.yellowdog.co/#/sdk/python-sdk.

## ⚠️ Critical: Generated vs. Hand-Written Code

**Significant portions of this SDK are auto-generated by the [`sdk-differ`](../sdk-differ/) tool and must not be manually edited.**

### Generated files (do not edit directly)

| Path | Generated from |
|------|---------------|
| `src/yellowdog_client/model/*.py` | Java model classes in `platform/public/model/` |
| `src/yellowdog_client/model/__init__.py` | All model classes (re-exports) |
| `src/yellowdog_client/*/[service]_client.py` | Java service client interfaces in `platform/public/service-clients-public/` |

These files are regenerated whenever the Java SDK (`platform` repo) changes. Any direct edits will be **overwritten** on the next generation run.

### Hand-written files (safe to edit)

All other files are hand-written and maintained in this repository:

- `src/yellowdog_client/platform_client.py` — Main `PlatformClient` entry point
- `src/yellowdog_client/*/[service]_client_impl.py` — Service client implementations
- `src/yellowdog_client/*/[service]_service_proxy.py` — HTTP proxy classes
- `src/yellowdog_client/common/` — Shared utilities (JSON, SSE, pagination, etc.)
- `tests/` — Test suite
- `pyproject.toml`, `tox.ini`, `scripts/` — Build and test configuration

### `# KEEP` sections in generated files

Some generated files contain hand-written code that must survive regeneration. This code is delimited by `# KEEP` comments:

```python
@dataclass
class TaskInput:
    # ... generated fields ...

    # KEEP
    @staticmethod
    def from_task_namespace(object_name_pattern: str, ...) -> 'TaskInput':
        """..."""
        ...
    # END KEEP (implicit — end of file)
```

The `sdk-differ` tool preserves all content after a `# KEEP` comment when regenerating a file. If you need to add permanent hand-written code to a generated file, place it in a `# KEEP` section at the end of the class. For new hand-written helpers that are substantial, prefer adding them to non-generated files.

## Repository Layout

This repo is one of three that must exist as siblings for the code generation pipeline to work:

```
parent/
├── sdk-differ/           ← Java code generator tool
├── platform/             ← Java source (read-only input to sdk-differ)
└── yellowdog-sdk-python/ ← This repo (generated output + hand-written code)
```

## Package Structure

**Package name:** `yellowdog_client` (in `src/`)

```
src/yellowdog_client/
├── platform_client.py          # Main entry point (PlatformClient facade)
├── client_collection.py
├── _version.py                 # Current version (e.g. 13.10.0)
├── py.typed                    # PEP 561 type marker
│
├── account/                    # Account, users, applications, groups, permissions
├── compute/                    # Compute requirements and cloud instances
├── scheduler/                  # Work requirements, tasks, worker pools
├── namespaces/                 # Namespace management
├── cloud_info/                 # Cloud provider metadata
├── images/                     # Machine image management
├── usage/                      # Allowances and usage tracking
│
├── common/                     # Shared utilities
│   ├── json/                   # JSON serialization/deserialization
│   ├── server_sent_events/     # SSE subscription support
│   ├── credentials/            # Authentication header providers
│   └── ...
│
└── model/                      # 300+ data model dataclasses (GENERATED)
```

### Architectural pattern

Each service module follows this layered pattern:

```
[Service]Client           (abstract ABC interface — often generated)
       ↓
[Service]ClientImpl       (hand-written implementation)
       ↓
[Service]ServiceProxy     (hand-written HTTP communication layer, extends Proxy)
```

The `PlatformClient` aggregates all 10 service clients via the facade pattern. Users construct it with `PlatformClient.create(schema, credentials)`.

## Development Setup

```bash
# One-time: create virtual environment and install dev dependencies
./scripts/setup

# Activate virtual environment
source .venv/bin/activate
```

All dependencies are declared in `pyproject.toml`. There is no separate `requirements.txt`.

**Python versions supported:** See `tox.ini` for the canonical list of supported versions.

The support policy is: the SDK targets the oldest actively supported Python version and drops a version as soon as it reaches End of Life (as defined by https://devguide.python.org/versions/). See the [internal standards document](https://yellowdog.atlassian.net/wiki/spaces/DEVT/pages/3167125505/Python+Version+Support) for details.

## Code Style and Conventions

### Type hints

All code must pass **strict mypy** targeting the minimum supported Python version (see `tox.ini`). Run the type checker with:

```bash
tox -e mypy
```

- Use `Optional[T]` (not `T | None`) for compatibility with the minimum supported Python version
- Use `from __future__ import annotations` in files that need forward references
- Avoid `Any` types
- Always parameterise generic types — write `dict[str, Any]`, `list[str]`, `tuple[int, ...]`, etc. Bare `dict`, `list`, `tuple` etc. will fail the `mypy` tox environment with a `Missing type parameters for generic type` error

### Dataclasses (model layer)

Model classes are `@dataclass` instances, often with `Optional` fields defaulting to `None`. Fields set by the server (not by the user) use `field(init=False)`. Example:

```python
@dataclass
class ComputeRequirement:
    namespace: str
    name: str
    tag: Optional[str] = None
    id: Optional[str] = field(default=None, init=False)  # server-assigned
```

### Abstract interfaces

Service client interfaces use `ABC` with `@abstractmethod`. Generated client files follow this pattern exactly — do not add concrete logic to them.

### HTTP proxy layer

Service proxies extend `Proxy` from `common/proxy.py` and communicate over HTTPS using the `requests` library. The `Proxy` base class handles authentication headers and retry logic.

### JSON serialization

The `jsons` library handles serialization/deserialization. Custom mappings live in `common/json/`.

### Naming

- Python naming conventions apply throughout (`snake_case` for methods/attributes, `PascalCase` for classes).
- The sdk-differ tool converts Java camelCase to Python snake_case automatically in generated files.

## Regenerating Generated Code

When the Java `platform` SDK changes (new models, updated client interfaces), regenerate the Python SDK using `sdk-differ`:

```bash
# From the parent directory containing all three repos
./sdk-differ/scripts/run
```

This runs the `sdk-differ` generator as a Docker container (or directly via Maven for local development — see [`sdk-differ` README](../sdk-differ/README.MD)).

After regeneration:
1. Review the diff to understand what changed
2. Ensure any `# KEEP` sections are still present and correct
3. Run tests to confirm nothing is broken: `./scripts/test`

## Building and Packaging

```bash
# Build wheel and source distribution
pip install build
python -m build
```

The package version is defined in `src/yellowdog_client/_version.py`. Bump it manually as part of your change — the CI/CD pipeline does **not** manage version bumps. Follow semantic versioning (minor for backwards-compatible additions such as new model fields, patch for fixes).
